# Чеклист захисту курсової

> **Призначення:** швидка перевірка перед захистом + список типових питань комісії.
> Використовуй цей файл в останні 2-3 дні перед здачею разом із [assignment.md](assignment.md) і [system-design.md](system-design.md).

---

## 1. Що має бути готово до захисту

- [ ] Репозиторій на GitHub відкривається без помилок
- [ ] `README.md` містить тему, стек, скриншоти, інструкцію запуску, demo-акаунти
- [ ] У репозиторії є `.env.example` без секретів
- [ ] Усі таблиці створюються через міграції або DDL без ручного дороблення в phpMyAdmin
- [ ] Є seed/demo-дані, щоб показати систему без ручного наповнення
- [ ] Мінімум 3 пов'язані сутності з FK реально працюють у UI
- [ ] Auth, ролі, CRUD, пошук, пагінація, upload зображень працюють без PHP warnings
- [ ] Пояснювальна записка готова у PDF
- [ ] ER-діаграма, схема маршрутів, 5-8 скриншотів інтерфейсу додані до ПЗ
- [ ] У коді немає `dd()`, `var_dump()`, debug-коментарів, тимчасових паролів, тестових URL

## 2. Швидкий прогін демо перед комісією

Порядок показу краще не міняти. Це дає зв'язний захист за 8-12 хвилин.

1. Відкрити головну сторінку і назвати тему системи.
2. Показати публічну частину: список, пошук, фільтр, сторінку деталей.
3. Увійти як звичайний користувач і показати основний user-flow.
4. Створити або змінити один запис у головній сутності.
5. Показати, що дані реально зберігаються і відображаються повторно.
6. Вийти і зайти як admin.
7. Показати admin CRUD, зміну статусу або модерацію.
8. Коротко показати БД: 3-5 ключових таблиць і FK.
9. Закрити демо поясненням архітектури: Model, View, Controller.

## 3. Мінімальний сценарій показу по типах систем

### Shop

1. Гість відкриває каталог.
2. Додає товар у кошик.
3. User оформлює замовлення.
4. Admin змінює статус замовлення.

### Booking

1. Гість переглядає послуги/майстрів.
2. User бронює слот.
3. Показати перевірку зайнятості часу.
4. Admin підтверджує або скасовує бронювання.

### Catalog

1. Гість переглядає каталог і фільтрує записи.
2. User додає в обране або резервує.
3. Admin створює новий запис і завантажує зображення.

### Community

1. Гість читає пости.
2. User створює пост або коментар.
3. Admin модерує контент або змінює статус.

### Internal

1. Увійти як user і показати свої записи.
2. Увійти як manager/admin і показати розширений доступ.
3. Показати dashboard, фільтри, зміну статусу або призначення відповідального.

## 4. Питання, які ставлять найчастіше

### По предметній області

- Чому ви обрали саме цю тему?
- Хто основний користувач системи?
- Яку реальну задачу вирішує ваш застосунок?

### По БД

- Які у вас головні сутності?
- Чому між цими таблицями саме такі зв'язки?
- Де у вас one-to-many, many-to-many?
- Які поля є обов'язковими і чому?
- Де використано `UNIQUE`, `FOREIGN KEY`, індекси?
- Чому таблиця не денормалізована або навпаки чому тут дублюється поле?

### По MVC та коду

- Де у вас Model, View, Controller?
- Чому SQL не повинен бути у view?
- За що відповідає контролер, а за що модель?
- Як організовано валідацію?
- Як реалізовано ролі та обмеження доступу?

### По функціоналу

- Який основний користувацький сценарій у системі?
- Що станеться при порожній формі або невалідних даних?
- Як користувач бачить результат своєї дії?
- Які додаткові модулі ви реалізували понад мінімум?

### По безпеці та якості

- Де хешується пароль?
- Як ви захищаєтесь від SQL injection?
- Як перевіряється доступ до admin-частини?
- Де зберігаються завантажені файли?
- Як відтворити проект на іншій машині?

## 5. Як відповідати на технічні питання

- Відповідай через конкретний файл або модуль: «це в `ProductController`, `Product` model і міграції `create_products_table`».
- Якщо показуєш ER-діаграму, одразу зв'язуй її з реальними таблицями в коді.
- Якщо питають «чому так», пояснюй через роль системи: швидкість пошуку, цілісність даних, зручність підтримки.
- Якщо чогось немає, краще прямо сказати «не встиг реалізувати, але запланував би так...», ніж вигадувати.

## 6. Що легко знижує оцінку

- [ ] Код запускається тільки «у мене на комп'ютері», без README і `.env.example`
- [ ] Показ тільки HTML без реальної БД-логіки
- [ ] Усі ролі перевіряються через `if ($user->id == 1)`
- [ ] Немає seed-даних, тому демо порожнє
- [ ] Комісія питає про зв'язки таблиць, а студент не може пояснити FK
- [ ] У ПЗ є скриншоти, але вони не збігаються з поточною версією проєкту
- [ ] Є кнопки/сторінки, які не працюють при живому показі
