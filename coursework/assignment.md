## 1. Мета роботи

Продемонструвати вміння спроєктувати, реалізувати і захистити повноцінний вебзастосунок з базою даних, аутентифікацією, ролями, CRUD-операціями, валідацією, зрозумілою архітектурою MVC та коректною документацією проєкту.

---

## 2. Технічний стек

Курсову можна виконувати **на одному з двох стеків**:

### Опція A — Laravel (рекомендовано, до 100 балів)

| Компонент     | Вимога                                           |
|---------------|--------------------------------------------------|
| Мова          | PHP 8.x                                          |
| Фреймворк     | Laravel 10/11                                    |
| БД            | MySQL / MariaDB                                  |
| ORM           | Eloquent (модель = клас в `app/Models/`)         |
| Шаблонізатор  | Blade                                            |
| Auth          | Laravel Breeze                                   |
| Версіонування | Git + GitHub                                     |

### Опція B — Vanilla PHP (на базі ЛР5, до ~80 балів)

- **Мова:** PHP 8.x
- **Архітектура:** власна MVC на основі ЛР4 з окремим Model-шаром
- **БД:** MySQL / MariaDB через PDO
- **Доступ до БД:** `classes/Database.php` як PDO singleton + класи в `models/`
- **Шаблонізатор:** PHP includes + `ob_start` (за аналогією з ЛР4 `renderDemoLayout`)
- **Auth:** власна форма входу/реєстрації + `password_hash`
- **Версіонування:** Git + GitHub

**Деталі вибору + порівняння можливостей** → [feature-catalog.md § 0](feature-catalog.md#0-%D1%81%D1%82%D0%B5%D0%BA-laravel-%D0%B0%D0%B1%D0%BE-vanilla-php-%D1%8F%D0%BA-%D0%BB%D1%805)

### ⛔ MVC — обов'язково на обох стеках

3 фізичні шари: **Model** (доступ до БД), **View** (шаблони), **Controller** (оркестрація). Жодного SQL у контролері/view, жодного echo/HTML у моделі. Кожна з 3+ сутностей — окрема модель. Деталі → [feature-catalog.md § MVC](feature-catalog.md#%E2%9B%94-mvc--%D0%BE%D0%B1%D0%BE%D0%B2%D1%8F%D0%B7%D0%BA%D0%BE%D0%B2%D0%B0-%D0%B2%D0%B8%D0%BC%D0%BE%D0%B3%D0%B0-%D0%B4%D0%BB%D1%8F-%D0%BE%D0%B1%D0%BE%D1%85-%D1%81%D1%82%D0%B5%D0%BA%D1%96%D0%B2)

## 3. Функціональні вимоги

### Обов'язкові модулі

| Модуль     | Деталі                                                          |
|------------|-----------------------------------------------------------------|
| Auth       | Реєстрація, вхід, вихід, ролі (адмін/користувач)                |
| CRUD       | Мінімум 3 пов'язані сутності (з FK)                             |
| Валідація  | Request-класи, зрозумілі повідомлення про помилки               |
| Міграції   | Всі схеми БД через `php artisan make:migration`                 |
| Пагінація  | 10-20 записів на сторінку                                       |
| Пошук      | По одній сутності (за назвою/іменем)                            |
| Зображення | Завантаження + `storage/app/public` + symlink                   |

### Додаткові модулі (опціонально, +бали)

- Email-сповіщення (SMTP / Mailtrap)
- REST API (Sanctum token auth)
- Багатомовність (`resources/lang`)
- Datatables / DataTables.net
- Експорт у PDF / Excel
- Інтеграція з зовнішнім API

## 4. Пояснювальна записка (ПЗ)

### Обсяг

Орієнтовно 30-45 сторінок друкованого тексту без урахування додатків. Формат A4, Times New Roman 14, інтервал 1.5, поля: ліве 30 мм, праве 10 мм, верхнє та нижнє 20 мм.

### Структура

1. Титульний лист
2. Завдання на курсову роботу
3. Зміст
4. Вступ (актуальність, мета, задачі)
5. Аналіз предметної області
6. Проєктування додатку
   - Діаграма БД (ER)
   - Схема компонентів
   - Схема маршрутів
7. Реалізація
   - Опис міграцій та моделей
   - Опис контролерів та маршрутів
   - Опис views та UI
8. Інструкція користувача (зі скриншотами)
9. Інструкція з розгортання
10. Висновки
11. Список використаних джерел
12. Додатки (лістинг ключового коду, повні скриншоти)

### Оформлення

Оформлення виконуйте за методичними рекомендаціями кафедри. Якщо окремий шаблон не видано, орієнтуйтеся на вимоги методички курсу, ДСТУ 3008:2015 для структури звіту та ДСТУ 8302:2015 для оформлення списку джерел.

## 7. Вибір теми

Тему обирайте за своїм варіантом з ЛР4-ЛР5 або узгоджуйте власну з викладачем. Найкращий сценарій — продовжити вже розпочату тему, щоб не починати з нуля БД, UI та бізнес-логіку.

### Приклади тем

- Інтернет-магазин книг, косметики або товарів для дому
- Система бронювання для готелю, стоматології або спортзалу
- Панель адміністрування для сервісу продажів, доставки або записів
- CRM для малого бізнесу
- Блог-платформа з модерацією
- Сервіс обліку витрат
- Система управління задачами
- Власна тема, погоджена з викладачем

> **Тема курсової = тема сайту з [ЛР4](../lr4/assignment.md) та [ЛР5](../lr5/assignment.md).** Рекомендується продовжувати свою тему, щоб використати напрацьований фронтенд та логіку.

## 9. Захист

Формат захисту: коротка презентація або структурований усний виступ, живе демо застосунку, пояснення архітектури та відповіді на питання. Орієнтовна тривалість: 10-15 хвилин. Від студента очікується не тільки показ інтерфейсу, а й розуміння БД, маршрутів, ролей, валідації та структури коду.

Перед захистом пройдіть [defense-checklist.md](defense-checklist.md) — там є готовий сценарій показу і типові питання комісії.

Структура виступу (10-15 хв):

1. Тема, мета, актуальність (1 хв)
2. Демонстрація ключових модулів (5-7 хв)
3. Архітектура: БД, маршрути, структура (2-3 хв)
4. Відповіді на питання
