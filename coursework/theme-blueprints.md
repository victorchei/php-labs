# Готові blueprints для типових тем курсової

> **Призначення:** закрити останню прогалину між абстрактними правилами і реальною темою студента.
> Якщо [functionality-flow.md](functionality-flow.md) відповідає на питання «який тип системи у мене?», то цей файл відповідає на питання «які саме сторінки, таблиці і модулі зібрати для моєї теми?».

---

## Як користуватись

1. Знайди тему, найближчу до своєї.
2. Візьми її як каркас, а не як жорстке обмеження.
3. Залиш ядро: 3+ сутності, ролі, головний user-flow, admin-блок.
4. Дрібні відмінності теми адаптуй через назви полів, статусів, довідників.

---

## 1. Книгарня

**Тип:** Shop
**Підходить для:** магазинів, пекарні, квіткової крамниці, зоомагазину, спортмагазину, аптеки

### Мінімум сторінок

- Головна
- Каталог книг
- Сторінка книги
- Кошик
- Оформлення замовлення
- Мої замовлення
- Login/Register/Profile
- Admin dashboard
- Admin: CRUD книг
- Admin: CRUD категорій
- Admin: список замовлень

### Мінімум таблиць

- `users`
- `categories`
- `products` або `books`
- `orders`
- `order_items`

### Добрі додаткові таблиці

- `authors`
- `book_authors`
- `reviews`
- `favorites`
- `addresses`

### Ролі

- `guest` — перегляд каталогу
- `user` — кошик, checkout, історія замовлень
- `admin` — керування товарами, категоріями, замовленнями, користувачами

### Основний сценарій захисту

1. Пошук книги в каталозі.
2. Додавання в кошик.
3. Оформлення замовлення.
4. Зміна статусу замовлення в адмінці.

### Типові поля

- `books.title`
- `books.price`
- `books.stock`
- `orders.status`
- `order_items.price_at_moment`

## 2. Стоматологія

**Тип:** Booking
**Підходить для:** ветклініки, барбершопу, фотостудії, автомийки, тренажерної зали, музичної школи

### Мінімум сторінок

- Головна
- Список послуг
- Список лікарів/майстрів
- Форма бронювання
- Мої бронювання
- Login/Register/Profile
- Admin dashboard
- Admin: CRUD послуг
- Admin: CRUD персоналу
- Admin: список бронювань

### Мінімум таблиць

- `users`
- `services`
- `masters` або `doctors`
- `bookings`
- `schedules` або фіксовані слоти в `bookings`

### Добрі додаткові таблиці

- `specializations`
- `rooms`
- `booking_status_logs`
- `reviews`

### Ролі

- `guest` — перегляд послуг
- `user` — створення та перегляд своїх бронювань
- `admin` — керування послугами, лікарями і статусами записів

### Основний сценарій захисту

1. Вибір послуги і лікаря.
2. Вибір вільного слоту.
3. Створення бронювання.
4. Підтвердження або зміна статусу в адмінці.

### Типові поля

- `bookings.booking_date`
- `bookings.booking_time`
- `bookings.status`
- `services.duration_min`
- `masters.specialization`

### Важливе бізнес-правило

Покажи, як система не дозволяє подвійне бронювання одного слоту для одного лікаря/ресурсу.

**Підходить для:** бібліотеки, кіностудії, вітринного каталогу послуг або експонатів

### Мінімум сторінок

- Головна
- Каталог експонатів
- Детальна сторінка експоната
- Пошук і фільтри
- Обране або резервування
- Login/Register/Profile
- Admin dashboard
- Admin: CRUD експонатів
- Admin: CRUD категорій/залів

### Мінімум таблиць

- `users`
- `categories`
- `items` або `exhibits`
- `favorites` або `reservations`

### Добрі додаткові таблиці

- `authors`
- `halls`
- `tags`
- `item_tags`

### Ролі

- `exhibits.era`

**Підходить для:** блог-платформи, новинного сайту з коментарями, контентної спільноти

### Мінімум сторінок

- Головна
- Список рецептів/постів
- Сторінка поста
- Створення поста
- Редагування свого поста
- Коментарі
- Login/Register/Profile
- Admin dashboard
- Admin: модерація постів
- Admin: модерація коментарів
- Admin: CRUD категорій

### Мінімум таблиць

- `users`
- `posts` або `recipes`

- `comments`

### Добрі додаткові таблиці

- `likes`
- `tags`
- `post_tags`
- `reports`

### Ролі

## 5. CRM малого бізнесу

**Тип:** Internal
**Підходить для:** панелі адміністрування, внутрішнього обліку, задач, заявок, клієнтів

### Мінімум сторінок

- Login
- Dashboard
- Список клієнтів
- Детальна сторінка клієнта
- Список угод/заявок/задач
- Створення і редагування записів
- Профіль
- Admin: користувачі
- Admin: довідники статусів/типів

### Мінімум таблиць

- `users`
- `clients`
- `deals` або `requests`
- `tasks`

### Добрі додаткові таблиці

- `departments`
- `statuses`
- `audit_log`
- `attachments`

### Ролі

- `user` — працює лише зі своїми записами
- `manager` — бачить команду або відділ
- `admin` — бачить усе і керує користувачами

### Основний сценарій захисту

1. Вхід у систему.
2. Створення клієнта або заявки.
3. Прив'язка задачі або зміна статусу.
4. Показ різниці доступу між user/manager/admin.

### Типові поля

- `clients.assigned_to`
- `deals.stage`
- `tasks.due_at`
- `users.role`

### Важливе бізнес-правило

Покажи, що закрита система не має гостьового доступу, а права реально розділені middleware або policy.

## 6. Як перетворити свій варіант у blueprint

Використовуй просту формулу:

1. Основний об'єкт системи: що продається, бронюється, читається або обробляється.
2. Довідник: що класифікує цей об'єкт.
3. Дія користувача: замовлення, бронювання, коментар, заявка, обране.
4. Рольова відмінність: що може admin або manager понад user.

Якщо ти можеш відповісти на ці 4 пункти, структура курсової вже визначена.

## 7. Що брати в ПЗ з цього файлу

- Назву типу системи.
- Перелік сторінок і модулів.
- Ключові сутності БД.
- Основний користувацький сценарій.
- Обґрунтування ролей.

Це напряму допомагає писати розділи «Аналіз предметної області», «Проєктування» і «Реалізація».
